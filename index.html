跨 AI 编程工具的项目级开发协议

让 Codex、Claude Code 和通用 Agent 按同一套流程开发。

Agent Pipeline Commander 用一个入口调度产品、后端、前端、测试和 UI 走查角色。它先建立项目画像，再按 feature 状态推进，所有交接都落到文档里。

如果发现项目画像与真实代码冲突，当前流程立即进入 project_rescan_required，重扫并确认后再继续。

不是多几个角色名，而是把 AI 开发变成可交接的流程。

每个角色只做自己的事，遇到不确定先确认；每一步都有产物、门禁和下一角色。

01

一个入口

用户只需要说“推进 feature”，Commander 根据 status.yaml 判断下一步角色和任务。

02

项目画像

首次扫描项目，后续开发前必读，开发后补充，让项目知识越用越准。

03

接口先行

有 Backend 介入时先出接口文档，确认后 Backend 和 FE 才开始同步开发。

04

分段验收

Backend 完成先测后端，FE 完成后按是否有设计材料决定是否先走 UI 验收。

05

扩测建议

Backend 和 FE 完成时都要给 Test 影响范围和建议测试点，帮助判断是否扩测。

06

跨工具安装

同一套 shared 资源适配 Codex、Claude Code 和支持 AGENTS.md 的工具。

真实开发流：先确认，再实现，最后全量验证。

full-stack 需求会经过需求澄清、接口文档、TODO 确认、分段验收和全量测试；没有设计材料时 UI 验收不会卡住前端测试。

Product 需求澄清 输出 brief、验收标准和测试覆盖方向。
Backend 接口文档 先写 api.openapi.yaml 和 backend/todo.md。
Confirm 开发前确认 使用者确认需求、接口文档和技术 TODO。
Build 分段开发 Backend / FE 可同步推进，并输出影响范围。
Test 全量测试 分段通过后执行全量测试，最终通知用户。

使用示范：把材料交给 Commander，它判断下一步。

不需要手动选择产品、后端、前端或测试角色。提供 feature、材料和意图，Commander 会归档输入、更新状态并输出下一步。

以项目为维度，不让 AI 每次重新认识仓库。

每个项目用 pipeline.project.yaml 绑定项目根目录、功能包目录和项目画像。画像错误时停止当前流程并重扫。

  • ✓ 首次使用先扫描。 生成 project-details.md，使用者确认后才能开发。
  • ✓ 每次开发前必读。 所有角色都以项目画像作为项目事实源。
  • ✓ 开发后反哺。 只补充可复用项目事实，不记录一次性需求细节。

一次安装，多个 AI 编程工具共用。

shared 目录是唯一事实来源，不同工具只安装自己的适配入口。

Codex 安装 pipeline-commander skill。
Claude Code 安装 /pipeline-commander slash command。
AGENTS.md 安装项目级 AGENTS.md 和资源目录。
All 同一命令安装全部适配器。

把项目知识、角色边界和测试门禁放进同一套协议。

适合长期维护的真实业务项目，也适合推广给不同 AI 工具的使用者。
