跨 AI 编程工具的项目级开发协议

让 Codex、Claude Code 和通用 Agent 按同一套流程开发。

Agent Pipeline Commander 用一个入口调度产品、后端、前端、测试和 UI 走查角色。它先建立项目画像，再按 feature 状态推进，所有交接都落到文档里。

不是多几个角色名，而是把 AI 开发变成可交接的流程。

每个角色只做自己的事，遇到不确定先确认；每一步都有产物、门禁和下一角色。

01

一个入口

用户只需要说“推进 feature”，Commander 根据 status.yaml 判断下一步角色和任务。

02

项目画像

首次扫描项目，识别前后端能力，后续开发前必读，开发后补充。

03

接口先行

有 Backend 介入时先出接口文档，确认后 Backend 和 FE 才开始同步开发。

04

分段验收

Backend 完成先测后端，FE 完成后按是否有可验收 UI 设计材料决定是否先走 UI 验收。

05

分层验证

基础检查、页面可达、截图核对和 Network 核对按变更类型分档执行。

06

跨工具安装

同一套 shared 资源适配 Codex、Claude Code 和支持 AGENTS.md 的工具。

真实开发流：先确认，再实现，最后全量验证。

full-stack 需求会经过需求澄清、接口文档、TODO 确认、分段验收和全量测试；没有可用于 UI 验收的设计材料时，UI 验收不会卡住前端测试。

Product 需求澄清 输出 brief、验收标准和测试覆盖方向。
Backend 接口文档 先写 api.openapi.yaml 和 backend/todo.md。
Confirm 开发前确认 使用者确认需求、接口文档和技术 TODO。
Build 分段开发 Backend / FE 可同步推进，并输出影响范围。
Test 全量测试 分段通过后执行全量测试，最终通知用户。

每个角色都有清晰边界，Commander 只在合适状态调起它。

使用者不用记角色名；这里说明它们分别能处理什么，以及会把哪些结论沉淀到功能包。

入口调度

Commander

读取项目画像和 status.yaml，判断当前功能下一步该由谁处理。

  • 识别 feature、材料、Bug 链接和提交意图
  • 按 phase / next 门禁调度角色
  • 发现项目画像冲突时停止并要求重扫
需求入口

Product

把需求材料变成可开发输入，不够清楚时先向使用者澄清。

  • 根据项目能力和需求意图建议 workflow
  • 整理 PRD、验收标准和需求边界
  • 拆分整合材料，归档 source-materials.md
  • 汇总接口、前后端 TODO 并发起开发确认
接口与服务

Backend

负责接口契约和后端实现，先把前后端协作边界讲清楚。

  • 输出 api.openapi.yaml 和 backend/todo.md
  • 实现服务、数据、权限和错误处理
  • 完成后给出影响范围和建议测试点
页面与交互

Frontend

负责前端 TODO、页面实现和接口联调；没有可验收 UI 设计稿时不强行卡 UI 流程。

  • 输出 frontend/todo.md 和 integration.md
  • 实现页面、状态、校验、接口接入和异常态
  • 记录 type-check、lint、页面可达、截图和 Network 核对
  • 开发期 UI 截图反馈可快修并记录
  • 说明改动范围、跳过 UI 原因和建议测试点
UI 走查

Designer

在有可验收 UI 设计稿或使用者单独要求时介入，检查实现与设计的一致性。

  • 读取 design/source.md 和 source-materials.md
  • 区分 UI 设计稿与产品示意图
  • 输出 Figma node、差异、期望 CSS/交互值和验证方式
  • 检查布局、视觉、交互、响应式和空态
  • 输出 design/ui-review.md，阻塞问题交回前端
质量验证

Test

负责分段测试、全量测试和 Bug 分诊，不替实现角色做通过结论。

  • 后端完成先测接口，前端完成再测页面流程
  • 根据影响范围决定是否扩大测试
  • 复核 UI 变更截图证据和接口 Network 证据
  • 真实送测 Bug 进入 bug_triage 后再分派修复
轻量提交

Commit

只在使用者明确要求提交时介入，整理提交范围并等待确认。

  • 生成 commit/notes.md 和 commit message
  • 中途 checkpoint 只提示已完成内容
  • Bug 修复后可补充外部平台备注

以项目为维度，不让 AI 每次重新认识仓库。

每个项目用 pipeline.project.yaml 绑定项目根目录、功能包目录和项目画像。画像错误时停止当前流程并重扫。

  • ✓ 首次使用先扫描。 生成 project-details.md，识别前端/后端能力，使用者确认后才能开发。
  • ✓ 每次开发前必读。 所有角色都以项目画像作为项目事实源。
  • ✓ 开发后反哺。 只补充可复用项目事实，不记录一次性需求细节。 
project:
  name: my-project

roots:
  project: .
  docs: lightweight-pipeline

knowledge:
  project_details: lightweight-pipeline/project-details.md

features:
  root: lightweight-pipeline/features

apps:
  backend:
    path: services/api
  frontend:
    path: packages/web

一次安装，多个 AI 编程工具共用。

shared 目录是唯一事实来源，不同工具只安装自己的适配入口。

Codex 安装 agent-pipeline skill，搜索位置更靠前。
Claude Code 安装 /pipeline-commander slash command。
AGENTS.md 安装项目级 AGENTS.md 和资源目录。
All 同一命令安装全部适配器。

把项目知识、角色边界和测试门禁放进同一套协议。

适合长期维护的真实业务项目，也适合推广给不同 AI 工具的使用者。
